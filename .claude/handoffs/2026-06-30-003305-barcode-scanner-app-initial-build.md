## Architecture Overview

Single Flutter codebase targeting Windows + Android + iOS. The critical design constraint: `mobile_scanner` and `google_mlkit_*` packages only support Android/iOS/macOS — NOT Windows. All scanning/OCR logic is hidden behind a `ScanEngine` abstract interface (`lib/core/scan_engine.dart`). Engine is selected at startup via `Platform.isWindows` and injected into the entire app via Riverpod. No platform branching anywhere else.

State management: `flutter_riverpod` (v3.3.2), using `Notifier` pattern. Navigation: `go_router` (v17.3.0). Local storage: `drift` ORM with `sqflite_common_ffi` for Windows compatibility. Product lookup: Open Food Facts (free, no API key). No accounts/cloud in MVP.

5. **sqflite_common_ffi**: Must call `sqfliteFfiInit()` and set `databaseFactory = databaseFactoryFfi` before `runApp()` on Windows/Linux. Done in `main.dart`.
6. **ResultsArgs**: Passed as `extra` in go_router push. Receive via `state.extra as ResultsArgs` in the route builder.
7. **KV key normalization**: `_keyAliases` map in `KvParser._normalizeKey()` maps abbreviations (EXP → Expiry, Batch → Batch/Lot). Add new aliases here.

## Decisions Made

| Decision | Options Considered | Rationale |
|----------|-------------------|-----------|
| Flutter (single codebase) | React Native, native per-platform | Single Dart codebase for Windows+Android+iOS |
| ScanEngine abstraction | Platform.isWindows checks everywhere | Forces clean separation; UI/logic never branch on platform |
| Riverpod (not Provider/Bloc) | Provider, Bloc, GetX | Riverpod 3.x is composable + testable; Notifier pattern replaces StateNotifier |
| Drift for history | sqlite3, sqflite, Hive | Drift has type-safe queries + codegen + works on all 3 platforms via sqflite_common_ffi |
| go_router (not Navigator 2) | Navigator 2 imperative, AutoRoute | go_router 17.x is official Flutter team package, declarative, clean deep-linking |
| In-memory product cache | Persistent disk cache, no cache | Simple, avoids stale data, acceptable for MVP scale |
| flutter_zxing for Windows | ZXing via platform channel, zxing4dart | flutter_zxing 2.x has FFI-based native decode + handles image decoding via readBarcodeImagePathString |
| OCR stub on Windows | Tesseract FFI | Tesseract requires bundling eng.traineddata + native lib; deferred to post-MVP |

## Potential Gotchas

- **`sqflite_common_ffi` init order**: Must be called BEFORE `runApp()`. It's in `main()` in `main.dart` — don't move it.
- **`MobileScanEngine` OCR via bytes**: The `bytes` path in `recognizeText()` uses `InputImageFormat.nv21` which is Android-specific camera format. For gallery/file images on mobile, use `filePath` not `bytes`.
- **Product lookup is online-only**: No offline fallback beyond the in-memory cache. If the network is unavailable, `lookup()` returns `null` silently and the KV result only shows OCR + barcode data.
- **`app_database.g.dart` must exist**: Without running `build_runner`, the project won't compile. It was already run this session, so the file exists. If it's missing, run: `dart run build_runner build`
- **Desktop OCR stub**: `DesktopScanEngine.recognizeText()` always returns `''`. Windows users will see barcode data + product lookup results only, no OCR key-value pairs from label text. This is intentional for MVP.
